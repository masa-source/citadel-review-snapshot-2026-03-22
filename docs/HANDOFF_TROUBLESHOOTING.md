# Citadel → Scout データ読み込み失敗の原因と対処

Citadel（Admin）から「Scout を起動して転送」したあと、Scout 側でデータの読み込みに失敗する場合の原因と対処をまとめます。

---

## データの流れ（Direct Handoff）

1. **Citadel** がバックエンドの `POST /api/sync/handoff` を呼ぶ → バックエンドがデータをステージングし、`ticketId` を返す。
2. **Citadel** が Scout を新しいタブで開く: `http://<Scoutのオリジン>/manage?ticket=<ticketId>&...`
3. **Scout** のデータ管理画面が読み込まれると、URL の `ticket` を使って `GET /api/sync/stage/<ticketId>` を**バックエンド**に送る。
4. バックエンドがステージング済み JSON を返し、Scout が IndexedDB にインポートする。

Scout は **Citadel から直接**データをもらうのではなく、**バックエンド**のステージング API から取得します。

---

## 根本原因: CORS（オリジン不一致）

### なぜ失敗するか

- ブラウザは **オリジン（スキーム + ホスト + ポート）** が違うと「クロスオリジン」として扱い、バックエンドが `Access-Control-Allow-Origin` で**明示的に許可したオリジン**以外にはレスポンスを渡しません。
- バックエンドのデフォルトは **`http://localhost:3000`（Scout）と `http://localhost:3001`（Admin）** のみ許可しています。
- **IP アドレスで Scout を開いた場合**（例: `http://192.168.1.100:3000/manage?ticket=...`）は、オリジンが `http://192.168.1.100:3000` になり、デフォルトの許可リストに**含まれません**。
- その結果、Scout からバックエンドへの `GET /api/sync/stage/<ticketId>` が CORS でブロックされ、**「データの読み込みに失敗」** のように見えます（ブラウザは「Failed to fetch」やネットワークエラーを返すことが多いです）。

### まとめ

| 状況                 | Scout のオリジン        | バックエンドの許可（デフォルト） | 結果                       |
| -------------------- | ----------------------- | -------------------------------- | -------------------------- |
| 同一 PC で localhost | `http://localhost:3000` | ✅ 含まれる                      | 成功                       |
| 他 PC や IP で開く   | `http://<IP>:3000`      | ❌ 含まれない                    | **CORS でブロック → 失敗** |

---

## 対処: ALLOWED_ORIGINS に Scout / Admin の実際のオリジンを入れる

バックエンドを動かしているマシンの **`.env`** で、Scout と Admin の**実際のアクセス先**を許可する必要があります。

- **同一 PC の localhost だけ使う場合**  
  → デフォルトのままでよい（`http://localhost:3000`, `http://localhost:3001`）。

- **IP や別 PC から Scout / Admin を開く場合**  
  → そのときに使う **Scout の URL のオリジン** と **Admin の URL のオリジン** を `ALLOWED_ORIGINS` に含めます。

例（サーバの IP が `192.168.1.100` の場合）:

```env
ALLOWED_ORIGINS=http://localhost:3000,http://localhost:3001,http://192.168.1.100:3000,http://192.168.1.100:3001
```

- Scout を `http://192.168.1.100:3000` で開く → `http://192.168.1.100:3000` を追加。
- Admin を `http://192.168.1.100:3001` で開く → `http://192.168.1.100:3001` を追加。

設定後は**バックエンドの再起動**が必要です。

---

## その他の要因（参考）

- **オフライン判定**  
  Scout のデータ管理画面では、`navigator.onLine === false` のときは Handoff 用の API 呼び出しをスキップします。  
  実際はオンラインなのにオフラインと誤判定されている場合は、ネットワークやブラウザの設定を確認してください。

- **API キー**  
  本番で `REQUIRE_API_KEY=true` にしている場合、Scout 用の `VITE_API_KEY_SCOUT`（Vite の環境変数）とバックエンドの `API_KEY_SCOUT` が一致している必要があります。不一致だと 401 で失敗します。

- **チケットの有効期限・ワンタイム**  
  ステージング API は「1 回取得したら削除」のワンタイムです。同じ `ticketId` で再度取得しようとすると 404 になります。  
  「データが見つかりませんでした」になる場合は、別タブで二重に開いた・再読み込みしたなどの可能性があります。

---

## Handoff は成功するがレポート編集・閲覧画面でフォームが空になる

**現象**: データ管理画面では件数が正しく表示され、レポート一覧でも件名などが表示されるが、一覧から「閲覧」または「編集」で開くとフォームがすべて空欄（新規作成と同じ）になる。

**再現手順の例**:

1. 受け側ノートPCで Citadel から Scout を起動（Direct Handoff）。
2. 他のページに遷移せず、オフライン用データをダウンロード（Handoff 完了）。
3. （任意）実際にオフラインにする（ネットワーク切断）。
4. レポート一覧から任意の報告書を「閲覧」または「編集」で開く → 新規作成フォームになる。

**想定される原因**:

- **クライアント遷移で id が渡らない**: 一覧→編集へ `router.push` で遷移すると、Service Worker のキャッシュや RSC 遅延により **id なしの編集ページシェル**（`/reports/edit/`）が返り、URL に `id=` が含まれないため編集画面が「新規」と判断される場合があります（オフライン時や Handoff 直後に顕著）。対策として、**一覧から編集・閲覧へは常に `window.location.assign` で完全遷移**しており、読み込み時の URL に id が含まれるため正しく表示されます。
- **古い IndexedDB / SW キャッシュ**: 上記以外に、受け側で古い IndexedDB やキャッシュが残っており一覧と編集で不整合が起きている場合もあります。環境をリセットすると解消することがあります。

**対処**（受け側の Scout を開いているブラウザで実施）:

1. **一覧→編集の遷移**: 現行版では一覧から編集・閲覧を開く際に常に完全遷移（`window.location`）を使うため、Handoff 直後やオフライン後でも正しく表示される想定です。不具合が続く場合は以下も試す。
2. **サイトデータの削除**: ブラウザの設定で Scout のオリジン（例: `http://<IP>:3000`）の「サイトのデータを削除」を行い、IndexedDB とキャッシュをクリアする。
3. **PWA の場合はアンインストール後に再インストール**: ホーム画面に追加したアプリを削除し、再度ブラウザから Scout を開いて「インストール」し直す。
4. その後、Admin から再度 Direct Handoff でデータを転送し、レポート編集・閲覧が正しく表示されるか確認する。

## チェックリスト

Citadel から Scout を起動したのにデータが読み込めない場合:

1. [ ] バックエンドの `ALLOWED_ORIGINS` に、**今 Scout を開いている URL のオリジン**（例: `http://<IP>:3000`）が含まれているか。
2. [ ] バックエンドを再起動したか。
3. [ ] ブラウザの開発者ツールの「ネットワーク」で、`/api/sync/stage/<ticketId>` が CORS や 4xx/5xx で失敗していないか。
4. [ ] （本番のみ）API キーが Scout とバックエンドで一致しているか。
