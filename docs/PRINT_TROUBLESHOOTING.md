# 印刷時の不具合の原因と対策

## 現象の整理

1. **一回目の印刷から Instrument Data が表示されない**
2. **編集後の二回目**: タグ番号など「一つしかない」項目が非表示になる
3. **使用部品が編集後に表示されない項目がある**
4. **すべてのデータがばらばらで、印刷されていないものもある**

※ 上記をテストで再現・原因特定するテスト: `apps/backend/tests/unit/test_placeholder_all_elements.py::test_all_placeholder_elements_resolve_and_diagnose`（バックエンドディレクトリで `python -m pytest` する場合は `tests/unit/...` で指定可）

---

## 想定される原因

### 1. Instrument Data が初回から表示されない

**原因: テンプレートの dataKey と実データの dataKey が一致していない。**

- `instrumentDataMap` は **DB の `data_key` の値**をそのままキーにしている。
- デモテンプレート（gen_templates）では `instrumentDataMap.温度` を参照しているが、実際のデータでは `data_key` が **"temperature"** など別の文字列になっていることが多い。
- キーが一致しないと `instrumentDataMap.温度` は存在せず、解決結果は空になり **何も表示されない**。

**対策:**

- テンプレート内の `instrumentDataMap.〇〇` の **〇〇 を、アプリで登録している「キー (dataKey)」と完全に一致させる**。
- 例: データが `dataKey: "temperature"` なら、テンプレートは `{{ inspectionDetails[0].instrumentDataMap.temperature.dataValue }}` とする。
- **設計台**: レポートを選択すると「Instrument Data を初回から印刷するには」の枠内に、そのレポートで使われている **dataKey 一覧**が表示されます。プレースホルダのキーをこのいずれかと完全一致させてください。「温度」「Volts」などの表示名では解決されません。
- デモ用テンプレートは、実データでよく使うキー（例: `temperature`）に合わせてある（後述の修正済み）。

---

### 2. 編集後にタグ番号など「一つだけ」の項目が非表示になる

**原因: 配列の「先頭 [0]」が、編集・保存のタイミングで変わるため。**

- テンプレートで `targetInstruments[0].tagNumber` のように **インデックス [0] で「1件目」を参照**している。
- 編集・同期・再保存により:
  - **sort_order の並び**が変わる
  - または **先頭の行が削除される**
  - または **別の対象計器が [0] に入る**
- その結果、「1件目」だった計器が [0] ではなくなったり、[0] が別の計器（タグ番号なしなど）になったりして、**タグ番号などが表示されなくなる**。

**対策:**

- **キーで参照する**: `targetInstrumentsById.<対象計器のID>.tagNumber` のように、**常に同じ対象計器を指す ID** で参照する。
- または、テンプレートで使う「1件目」の意味を、運用で固定する（並び順を変えない・先頭を消さない）か、キー参照に切り替える。

---

### 3. 使用部品が編集後に表示されない項目がある

**原因: テンプレートが `usedParts[0]` や `usedPartPrimary` で参照しており、編集・同期で配列順が変わると [0] が別の実体を指すため。**

- 編集・同期・再インポートで **used_parts の sort_order や配列の並び**が変わると、`usedParts[0]` が別の部品を指す。
- その結果、「1件目」だった部品が [0] ではなくなったり、[0] の part が期待と違うため **表示されない**ように見える。
- また **usedPartsById のキーは DB 採番の id**。importer は UsedPart の id を入力から取らず採番するため、Scout から送る id とコンテキストの id が一致しない（上書きインポートで再作成されると id が変わる）。

**対策:**

- **キー参照を使う**: コンテキストに存在する id（`load_report_context` の結果の `usedParts[].id`）で `usedPartsById.<id>.part.name` のように参照する。テンプレート作成時は「このレポートの使用部品一覧から id を選ぶ」運用にする。
- または「1件目」でよい場合は `usedPartPrimary` を使うが、編集後に並びが変わると別の部品になる点を許容する。

---

### 4. データがばらばら・一部が印刷されない

**原因: 配列の並びが変わり、[0] [1] が指す「実体」がずれるため。**

- テンプレートが `reportWorkers[0]`, `reportWorkers[1]`, `targetInstruments[0]` など **インデックスでセルと実体を対応**させている。
- 編集・同期・再インポートで **sort_order や配列の並び**が変わると:
  - 同じセルに、**別の作業者・別の計器**の値が入る → 「ばらばら」に見える。
  - 元々 2 件だったところが 1 件になると、[1] が存在せず **空になる** → 「印刷されていない」ように見える。

**対策:**

- **キー参照に統一する**（推奨）:
  - 例: `reportWorkersByWorkerId.<workerId>.worker.name`
  - 例: `targetInstrumentsById.<targetInstrumentId>.tagNumber`
  - 例: `reportClientsByCompanyId.<companyId>.company.name`
- こうすると、「どのセルに誰の／どのデータを出すか」が **ID で固定**され、並びが変わっても同じセルに同じ実体の値が入る。

## まとめ

| 現象                                   | 主な原因                                                                                           | 対策                                                            |
| -------------------------------------- | -------------------------------------------------------------------------------------------------- | --------------------------------------------------------------- |
| Instrument Data が初回から表示されない | instrumentDataMap のキーが実データの dataKey と不一致（例: 温度 vs temperature, Volts vs voltage） | テンプレートのキーを実データの dataKey に合わせる               |
| タグ番号などが非表示                   | [0] の実体が編集で変わる／消える                                                                   | キー参照（targetInstrumentsById 等）を使う                      |
| 使用部品が編集後に表示されない         | usedParts[0] / usedPartPrimary で参照しており、編集で配列順が変わると [0] が別の部品を指す         | usedPartsById.<id> で参照（id はコンテキストの usedParts[].id） |
| データがばらばら・一部だけ印刷されない | 並び替えで [0][1] の指す実体が変わる                                                               | キー参照に統一し、ID で一意に参照する                           |

データ設計ではなく、**テンプレートの参照方法（キー vs インデックス）と、キーの文字列一致**が原因です。上記の対策で解消できます。

---

## 報告書の「更新の仕方」と ID 保持

以前のバージョンでは、上書きインポート時に関連データの ID を再採番していましたが、現在は **Scout から送られた ID をそのまま使用 (UPSERT)** するように統一されています。これにより、テンプレート内で ID を使ったキー参照（`usedPartsById.<id>` 等）を行っても、編集・同期の前後で参照が壊れることはありません。

---

## 事例: 「保存してシタデルに送っただけ」で作業者(1件目)が別人になった

**現象**: 開発版の [DEMO] 定期点検報告書 No.1 で、Scout で行った操作は「保存してシタデルに送っただけ」なのに、作業者(1件目)が [DEMO] 山田 太郎 から [DEMO] 鈴木 一郎 に変わった。

**考えられる状況:**

1. **「作業者(1件目)」は sort_order が最小の作業者**  
   テンプレートが `reportWorkerPrimary` や `reportWorkers[0]` で参照しているため、サーバー側の `report_workers` の **sort_order 0 の人が誰か**で表示が決まる。

2. **Scout で保存・送信すると、Scout のフォーム上の並びがそのまま送られる**  
   フォームの並びは IndexedDB の `reportWorkers` を **sortOrder でソートした結果**。送信ペイロードの配列順と各 `sortOrder` で上書きされる。

3. **並びが変わった要因の例**
   - **バックエンドのエクスポート順**: 同期でサーバー→Scout に渡すとき、`report.report_workers` の**配列順が sort_order でソートされていなかった**（ORM の返す順が id 順など）。各要素の `sortOrder` は正しくても、Scout が受け取る配列順が「見た目の1件目」とずれていた可能性。
   - **サーバー側の元データ**: デモデータや前回同期の結果として、もともと 鈴木 一郎 が sort_order 0 だった。以前「山田が1件目」だったのは別レポートや古い状態だった可能性。

### 対策と推奨される参照方法

- **エクスポートでのソート順保証**: バックエンドの同期用エクスポートでは、各子テーブル（作業者、計器等）を必ず `sort_order` 昇順でソートして出力します。これにより、インデックス参照（`[0]`）の安定性が高まっています。
- **ID 参照の推奨**: 並び順の変更に強いテンプレートを作成するには、`reportWorkersByWorkerId.<workerId>.worker.name` のように **ID で参照**することを推奨します。
