# バックエンド テスト設計ガイドライン

テストコードの責務を明確にし、ディレクトリごとにテストを適切に配置するためのガイドラインです。

## ディレクトリと責務

### `unit/`
- **目的**: 外部依存（DB、ファイルシステム、ネットワークなど）を持たない、純粋な関数やクラスのテスト。
- **ルール**: 
  - フィクスチャとして `db_session` を使用・注入することは**禁止**です。
  - シリアライズ処理（JSON変換）、日付計算、純粋なユーティリティ関数などがここに配置されます。

### `services/`
- **目的**: ビジネスロジックのテスト。
- **ルール**: 
  - DBセッション (`db_session`) を直接注入し、データの永続化、トランザクションの振る舞い、外部キー関係の解決など、**内部状態の深い検証**を行います。
  - インポート・エクスポート処理や、複雑なドメインロジックの振る舞いを保証します。

### `api/`
- **目的**: FastAPIのエンドポイントテスト。
- **ルール**: 
  - `httpx.AsyncClient` を使用し、HTTPステータスコード、リクエスト/レスポンスのスキーマ、認証・認可の動作などを検証します。
  - API境界の振る舞いに集中するため、**DBの内部状態の深い検証（`db_session` を使った複雑なアサートなど）は行いません**（必要な場合は `services/` で検証します）。
  - ※ただし、テストデータの事前セットアップ（Fixture）としてDBセッションを使用することは許可されます。

### `infrastructure/`
- **目的**: DBインフラ、外部ライブラリのラッパーなどのインフラ層の検証。
- **ルール**: 
  - カスタムのデータ型（例: `EncryptedString` や特殊なJSONエンコーダ等）の実DBを介した振る舞いや、ストレージ・ネットワーク通信のラッパー関数のテストなどを配置します。

## AIプロンプト評価テストの実行方法

AIのプロンプト精度を検証する **AI評価テスト（LLM Evaluation）** は、ローカルで AI（例: Qwen3.5）が起動している場合にのみ手動で実行してください。CI では実行しません（時間がかかり、AI が利用できないため）。

**実行例:**

```powershell
# プロジェクトルートから実行（venv を自動解決するため pnpm test:backend 経由を推奨）
pnpm test:backend -- tests/services/test_ai_extractor_eval.py -v -m ai_eval

# または venv を明示する場合（必ずプロジェクトルート Cwd から実行すること）
.\venv\Scripts\python.exe -m pytest apps/backend/tests/services/test_ai_extractor_eval.py -v -m ai_eval
```

必要に応じて `--timeout=120` などを付けてタイムアウトを延長してよい。
